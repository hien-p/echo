2026-05-12 fix

Submissions = 0 + lag — unify queryEvents + drop redundant aurora

After the KPI strip mounted above the triage queue, submissions count dropped to 0 even though 7 forms loaded. Root cause: KpiStrip's local queryEvents used a MoveEventField-only filter while CrossFormDashboard used the proven All [MoveEventType + MoveEventField] shape. KpiStrip ran first → its broken queryFn populated the shared cache with empty data → CrossFormDashboard read 0 from cache. Exported queryEventsByFormId + the SubmissionEvent type from CrossFormDashboard, replaced the local copy in KpiStrip. Both components now share one queryFn implementation as well as one cache. Also dropped the second AuroraBlur shader on the KPI section (the hero already runs one) — two simultaneous WebGL canvases were the source of the lag.

2026-05-11 feat

Walrus image proxy — embed-anywhere image / GIF / video

Walrus aggregators serve blobs with x-content-type-options: nosniff and no content-type header, so Chrome silently refuses to render them in <img>. New edge route /api/walrus/blob/[id] fetches the blob, sniffs magic bytes (PNG / JPEG / GIF / WebP / MP4 family / WebM / OGG / AVI / HEIC / WAV / PDF), and re-emits with the correct content-type + cache-control: immutable, max-age=1y. Markdown editor inserts absolute URLs pinned to the current origin so the markdown is portable — paste a Walrus screenshot into a GitHub README, Notion, blog, anywhere — the embed still works because access-control-allow-origin: * lets the proxy serve any origin. Drop / paste / pick now accepts videos too; videos get a #video URL fragment hint so MarkdownView swaps <img> for <video controls preload="metadata" playsInline> without needing rehype-raw.

2026-05-11 fix

Three load-bearing bugs squashed under the takeover

500 on every /forms/[id] traced to a known @cloudflare/next-on-pages 1.13.x bug: ReferenceError: async__chunk_<id> is not defined when React 19 + edge runtime + nested dynamic({ ssr: false }) wrap a module graph past a certain size. Local dev never reproduced. Removed the redundant dynamic wrapper in FormViewerClient (parent was already "use client") and the bug evaporated. ~150px-wide content column with every word on its own line turned out to be this project's Tailwind 4 theme shadowing t-shirt keys (max-w-2xl resolves to --spacing-2xl, ~px not 42rem) — the globals.css comment literally calls this out. Switched to arbitrary widths (max-w-[680px] etc.). Image upload silently wiped on completion was a stale-closure race in MarkdownEditor.uploadAndInsert: the async resolve ran onChange(value.replace(placeholder, final)) against the closure-captured pre-insert value, so the placeholder we'd just inserted got overwritten with the original value. Fixed via a latestValueRef mirrored on every render.

2026-05-09 feat

Real m-of-n threshold via on-chain ApprovalWitness

Threshold tier now does what it says. Previously "k of n" was OR-of-N (any single cap holder decrypts) — k was a label, not a check. New flow: each admin posts a shared ApprovalWitness object via echo::form::post_approval; once ≥ k unique-signer witnesses exist for the form's Seal identity, seal_approve_threshold_m_of_n takes the witness vector, asserts (a) count ≥ k, (b) each binds to this form + identity, (c) signers all distinct. Anyone can finalize decrypt at that point — it's a votes-to-release primitive, not an ongoing access gate.

FormBuilder restored the Required approvals (k) input (1 to total admins). FormAdmin polls ApprovalPosted events every 8s and shows "k/n approvals · waiting for X more". When you hold a cap and haven't signed yet, an "Approve decrypt" button appears. Once the k-th witness lands, the existing Decrypt / Reveal-all / Index paths switch from seal_approve_threshold to seal_approve_threshold_m_of_n with the collected witness IDs. Backward compat: forms with k=1 still use the old single-cap path.

Trade-off, communicated in the UI: witnesses live on chain forever (Seal calls seal_approve_* via dry-run, so the in-function object::delete never persists). So "k of n approved once" → data is decryptable forever by anyone who knows the witness IDs. Right shape for board votes / sealed bid reveals / DAO disclosure releases. Wrong shape for ongoing per-read authorization (would need re-encryption per session).

Demo: FORM=multisig creates a 2-of-3 form with two extra admin caps. New package id 0x47a7…5612 (re-published to testnet — old forms on 0x76b0…bc37 are orphaned). Move tests added covering (a) post_approval mints witness with right signer, (b) k unique witnesses pass seal_approve, (c) below-k aborts with EInsufficientApprovals (8), (d) duplicate signer aborts with EDuplicateSigner (11). 22/22 Move tests passing.
